# v0.7.11 Catalog Contexts Design

## Goal

Stabilize the existing exercise catalog without free expansion. The app must preserve 305 catalog entries and 294 visible exercise names, distinguish repeated exercise contexts, validate every entry description, and reduce fragile filtering that depends only on string inference.

## Current Findings

The catalog source of truth is `SeedData.exercisesByGroup`. It contains 305 entries and 294 unique names. The database seed currently looks up default exercises with `name = ?`, so repeated names collapse to one row and later refreshes can overwrite the row with the last context. This loses contexts for entries such as `Face pull no cabo`, `Pullover com halter`, `Good morning sem carga`, `Curl inverso`, and `Drills de guarda`.

Exercise descriptions are generated by `ExerciseCatalogDetailService`. The service has useful specific branches for important exercises, but many entries still fall through to family helpers such as `_mobilitySteps`, `_cardioSteps`, `_dumbbellSteps`, and `_bodyweightSteps`. v0.7.11 must keep helpers as internal building blocks only, with zero entries depending only on an unadapted fallback.

Filtering still uses `contains` in several places. v0.7.11 will not rewrite the entire architecture, but will add explicit catalog entry keys and context keys, use those keys where available, and document any remaining `contains` as fallback.

## Design

Add explicit catalog identity fields to `Exercise` and the `exercises` table:

- `exercise_key`: stable key for the movement name, for example `face_pull_no_cabo`.
- `context_key`: stable key for the context/group path, for example `ombros`.
- `catalog_entry_key`: stable composite key, for example `face_pull_no_cabo__ombros`.

Update seeding and refresh logic so default exercises are identified by `catalog_entry_key` or by `name + muscle_group` when migrating older databases. New installs and upgrades must preserve all 305 entries. Existing user data is preserved because old exercise IDs are not deleted; missing context rows are inserted.

Create a small catalog context service that can derive all 305 catalog entries from `SeedData.exercisesByGroup`. It provides stable keys, counts, duplicate context reports, and per-entry details. The runtime model remains compatible with existing UI screens because each context row is still an `Exercise`.

Descriptions are validated per catalog entry. For entries that currently fall through to family-only helpers, v0.7.11 will generate exercise-specific beginner text that names the exercise, equipment, target focus, position, movement, breathing, amplitude, common errors, and safety. The audit report records one line per entry and marks every final state as `OK VALIDADO`, `REESCRITO`, `METADATA CORRIGIDA`, or `REESCRITO E METADATA CORRIGIDA`.

## Testing

Tests will be added before implementation for:

- 305 seed entries and 294 unique names.
- Database seeding preserving duplicate contexts instead of collapsing by name.
- `Face pull no cabo`, `Pullover com halter`, `Good morning sem carga`, `Curl inverso`, and `Drills de guarda` contexts.
- All 305 catalog entries having non-empty details, no forbidden generic phrases, beginner-readable steps, equipment, mistakes, and safety.
- Zero entries depending only on family fallback helpers.
- Ambiguous names either renamed, clearly equipped, or justified by metadata.
- Key filters such as `Braços completo`, `Peito completo`, `Costas completo`, modality-specific cardio, mobility, and recovery.
